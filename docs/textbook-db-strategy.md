# 参考書 DB 戦略

調査日: 2026-05-14

---

## Studyplus の仕組み (調査結果)

**データ取得元: Amazon + バーコード UGC が中核（推定）**

公開情報から分かった範囲:

- 「Amazonで売っている書籍なら全て登録可能」と公式が明言 → Amazon PA-API (Product Advertising API) か Amazon カタログデータを書籍 DB のベースに使っていると推定される
- ユーザーがバーコード (ISBN) をスキャン → 書籍メタデータを取得してテキスト登録 → UGC として蓄積する仕組み
- **Studyplusブックコード** という出版社向け有償連携スキームがある。対応教材には CSVコード を発行し、生徒が専用コードで教材を登録できる。清水書院・旺文社・学研など主要出版社が参加済み
- 登録冊数の公開情報は見当たらないが、Amazon JP の書籍点数（200万冊超）をカバーできる構造

**信頼度: 中（技術詳細は非公開、行動から逆算した推定）**

## 競合の動き

### Clear (旧 Clearnote)
- ノート共有 UGC アプリ。「科目 × 単元」タグで整理
- 参考書 DB を独自構築している証拠は見当たらず、ノート側のタグが参考書名と対応する形で自然発生的に蓄積されている模式
- 書籍メタデータを DB として保持するより「ノート ≒ 参考書ページ」として機能している

### 旺文社 StudiCo / 学びの友
- 旺文社が自社刊行 100 冊以上を自前でデジタル管理し、アプリから直接連携
- 外部に目次 API を公開しているわけではなく、自社 closed な DB
- 学研・増進堂・受験研究社も類似の自社 closed 構造と推定

### 過去問道場
- 試験別の問題 DB が中心。参考書 DB は持っていない

## データソース別の可能性

| ソース | 取得できる主な項目 | 量（日本書籍） | コスト | 目次 |
|-------|-------------------|--------------|-------|------|
| NDL Search API | ISBN / タイトル / 著者 / 出版社 / ページ数 (dc:extent) / 件名 | 約 1,200 万件 | 無料 | なし |
| openBD | ISBN / タイトル / 著者 / 表紙画像 / 価格 / 目次 (onix TOC) | 約 200 万件 | 無料 (上限なし) | 一部あり (数%程度) |
| Google Books API | タイトル / 著者 / 概要 / tableOfContents / 表紙 | 不明 | 無料 (1,000 query/日) | ほぼ空 (日本語は特に) |
| Amazon PA-API | タイトル / 著者 / 表紙 / 価格 / カスタマーレビュー | Amazon 出品全点 | 売上報酬型 (厳格) | なし |
| スクレイピング | サイト依存 (Amazon・出版社サイト) | 制限なし(技術的) | 人件費・インフラ費 | サイト依存 |

### 各ソースの補足

**NDL Search — dc:extent フィールド**
- レスポンス例: `"340p ; 21cm"` のような自然言語文字列で返却される
- 数値として直接パースできないケースが多い。正規表現で抽出は可能だが全冊ではない
- ISBN 検索の精度は高く、一括取得 (OAI-PMH) も可

**openBD — TOC フィールド**
- ONIX 準拠の `toc` フィールドが存在するが、**データが入っている書籍は全体の 5% 未満と推定**（調査コミュニティの複数報告より）
- 受験参考書で目次が入っているのは、出版社が ONIX データを正確に登録した書籍に限られる
- 表紙画像・価格・ISBN はほぼ全件取得可能で実用的

**Google Books — tableOfContents**
- 日本語書籍は英語書籍と比べてデータ充填率が著しく低い（未検証の報告が多数だが、直接確認可能）
- 無料枠は 1,000 クエリ/日 = 大量取り込みには不向き

**Amazon PA-API**
- アソシエイトプログラムへの参加が前提で、売上報酬の発生が条件
- 利用規約でデータの永続保存・DB 構築を禁止している（法的リスク大）

## 目次データの現実

| 取得手段 | 日本の受験参考書で使える割合 | 備考 |
|---------|--------------------------|------|
| openBD TOC フィールド | ~5% 未満 | 出版社の ONIX 登録次第 |
| Google Books tableOfContents | ~1-3% | 日本語書籍は特に空が多い |
| NDL Search | 0% | 目次フィールド自体なし |
| スクレイピング (Amazon 商品ページ) | ~40-60%? | 規約違反リスク大 |
| AI 生成 (Claude / GPT-4) | 主要 200 冊なら 90%+ 精度 | 著作権上の問題なし (生成物であるため)、事実確認必要 |
| 出版社直連携 | 100% (連携先のみ) | 旺文社・学研・Z会 など交渉次第 |

**AI 生成目次の実現性について**
- チャート式・青チャ・フォーカスゴールド・スタディサプリ的な「定番参考書」は発行部数が多く、AI が内容を十分に学習済みと考えられる
- 「第1章: ○○ / 第2章: ○○」レベルの大目次なら高精度で再現可能と推定 (未検証)
- 細かい「節・項」レベルの正確性は要実験。人間によるレビューを挟む設計が必要
- 著作権: 目次自体は事実の列挙であり、日本著作権法上は保護対象外とする解釈が一般的だが、出版社との関係上確認を推奨

## Testall の推奨戦略

### Phase A — 量で基盤構築 (1,000〜2,000 冊) ※MVP フェーズ

1. **NDL Search 一括取得 (OAI-PMH)** で受験参考書カテゴリの ISBN リストを収集
2. **openBD バルク API** で各 ISBN のメタデータ（タイトル・著者・表紙・価格）を取得
3. ISBN が一致しない書籍は Google Books API で補完
4. このフェーズは **0 円 / 0 法務リスク** で 1,000 冊以上のカバーが可能

**想定工数: エンジニア 1 人 × 1 日**

### Phase B — 質の向上 (主要 200 冊) ※コア体験に直結

1. 受験生の使用頻度上位 200 冊をリストアップ (studyplus 科目別一覧 + 各予備校サイト参照)
2. **Claude で目次・強み・弱み・対象レベル** を生成
3. 人間 (大学生バイト or 在校生) が 1 冊あたり 15 分でレビュー・修正
4. コスト試算: Claude API ≒ ¥2,000〜¥5,000 / 200 冊 + レビュー人件費

**このフェーズで Testall の差別化コンテンツが生まれる**

### Phase C — UGC で拡張

1. バーコードスキャン機能でユーザーが手持ち参考書を追加できるようにする
2. ユーザーが「使ってみた感想・学習法」を投稿 → Phase B データを補強
3. 一定 UGC が蓄積したら **出版社に直接アプローチ** して公式データ連携を提案（Phase B で構築した実績を交渉材料にする）

## リスク・注意

| リスク | 深刻度 | 対策 |
|-------|-------|------|
| Amazon PA-API でのデータ永続保存 | 高 | **絶対に使わない**。利用規約で DB 構築禁止が明記 |
| Amazon / 出版社サイトのスクレイピング | 高 | robots.txt と利用規約を要確認。商業サービスでは原則 NG |
| openBD データの精度 (ISBN ミス等) | 低〜中 | 表示前に人間レビューを挟む |
| AI 生成目次の誤り | 中 | 必ず人間レビューを挟み「AI 生成・要確認」ラベルを表示 |
| 著作権 (目次の転載) | 低 | 目次は著作権保護対象外の解釈が一般的だが、AI 生成と明示すれば回避しやすい |
| Google Books 1,000 query/日 制限 | 中 | NDL + openBD で 8 割補完できるため補助利用に留める |
